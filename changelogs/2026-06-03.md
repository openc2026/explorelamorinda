## Notes

- **PowerShell `Get-Content` lies about UTF-8 content on Windows-1252 codepages.** When grepping the site files for `Buying or selling`, the console showed `�?"` glyphs where a U+2014 em-dash should be — which looks identical to the U+FFFD defect yesterday's pass fixed in the things-to-do cluster. It is not the same thing: U+FFFD is a single code point (the literal Unicode replacement character) that Hugo passes through and that browsers render as a question-mark-in-a-diamond; `�?"` displayed by PowerShell is the three-byte UTF-8 encoding of U+2014 (E2 80 94) being decoded as Windows-1252 by the default console codepage and printed as three separate visible glyphs. The first case is a real bug; the second is a console-display artifact and the file is fine. **Rule for future passes**: when a scan turns up suspicious glyphs, immediately re-check with `[System.IO.File]::ReadAllText($path).Contains([char]0xFFFD)` (which returns the actual `bool` for "U+FFFD present in the parsed string") before assuming a defect exists. Saved ~10 minutes of would-be cluster-wide edits today.
